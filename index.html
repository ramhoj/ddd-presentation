The domain model defines the epicenter‚Äîthe behaviors and nouns of your business.

Rich, coherent domain objects

app/models/card.rb 
class Card < ApplicationRecord
  include Assignable, Closeable, Golden, Postponable,
    Searchable, Taggable, Triageable, Watchable

  belongs_to :board
  belongs_to :creator, class_name: "User"

  has_many :comments, dependent: :destroy
  has_rich_text :description

  scope :golden, -> { joins(:goldness) }
  scope :latest, -> { order(last_active_at: :desc) }
end

Concerns organize traits and roles. The model stays readable and coherent.

‚Üê ‚Üí

Traits as concerns

app/models/card/golden.rb 
module Card::Golden
  extend ActiveSupport::Concern

  included do
    has_one :goldness, dependent: :destroy
    scope :golden, -> { joins(:goldness) }
  end

  def golden?
    goldness.present?
  end

  def gild
    create_goldness! unless golden?
  end

  def ungild
    goldness&.destroy
  end
end

Things that belong together, live together. Easy to understand, easy to maintain.

Chapter 2

A Simpler Way

Code that reads like prose

‚Üê ‚Üí

Service objects are not meant to implement your business logic. They are meant to orchestrate your domain objects.

Jorge Manrubia, 37signals

‚Üê ‚Üí

Boilerplate obscures intent

With a service object

# In the controller
CardSendBackToTriage.call(card)

# Separate class required
class CardSendBackToTriage
  def initialize(card)
    @card = card
  end

  def call
    @card.send_back_to_triage
  end
end

Directly on the domain

# In the controller
card.send_back_to_triage


# That's it.
# The behavior lives
# in the domain model
# where it belongs.

The service object wraps one line of code with a whole class.
The boilerplate draws attention away from the business logic.

‚Üê ‚Üí

Semantic names over generic patterns

CardGildingService.call(card) ‚Üí card.gild
ResumeCardOperation.new(card).execute ‚Üí card.resume
SignupService.call(params) ‚Üí signup.create_identity

Generic verbs like call and execute say nothing about intent.
Domain methods read like prose: "The card is gilded. The signup creates an identity."

Chapter 3

Orchestration

Controllers, jobs, and plain Ruby objects

Controllers orchestrate the domain

app/controllers/cards/closures_controller.rb 
class Cards::ClosuresController < ApplicationController
  include CardScoped

  def create
    @card.close
  end

  def destroy
    @card.reopen
  end
end

The controller connects domain objects with the external world.
This is the role of a service object‚Äîwe just don't need a separate class.

Jobs work the same way: thin orchestration of domain logic.

module Card::Stallable
  extend ActiveSupport::Concern

  included do
    has_one :activity_spike, class_name: "Card::ActivitySpike"
    after_update_commit :detect_activity_spikes_later
  end

  def stalled?
    open? && last_activity_spike_at < 14.days.ago && updated_at < 14.days.ago
  end

  def last_activity_spike_at
    activity_spike&.updated_at
  end

  def detect_activity_spikes
    Card::ActivitySpike::Detector.new(self).detect
  end
end

Concerns don't replace object-oriented design‚Äîthey complement it.

Key Takeaways

  • Place the domain model at the center of your architecture
  • Use concerns to organize traits‚Äîthings that belong together, together
  • Prefer semantic names over generic patterns like call
  • Controllers and jobs orchestrate; they don't hold business logic
  • Plain Ruby objects + ActiveModel work beautifully as domain models
  • Prioritize what's coherent for humans over theoretical purity

These are suggestions offered humbly‚Äînot mandates.
What matters is code that's easy to understand and maintain.
